# Adversarial Improvement Review — P1/P2 Ontology Hardening

## 1. Verdict
P1/P2 staging is directionally correct — modular build, relational champion preserved, PROV-O/OWL-Time named, promotion gated by `DecisionFeatureActivation` — but it is under-specified exactly where trading-safety lives, and it carries a dependency-order inversion. **Top risk: temporal/bitemporal cutoff safety and correction lineage are named as vocabulary but never mechanized (no cutoff predicate, no `wasRevisionOf` shape), so H12's cutoff-leakage tests have nothing to assert against and H14 can promote graph growth that "conforms" the same way the empty P0 vocabulary did.** The current order (H8 evidence before H9 events; H10 markers before H11 portfolio) forces forward-references or duplicated temporal predicates that break manifest hashing.

## 2. Confirmed strengths
- Preserves the load-bearing invariant: relational snapshot-scoped retrieval stays the permanent champion; Fuseki/Neo4j are rebuildable challengers only; no number crosses the research seam.
- ADR-0005 time-scoped `IssuerRole` is correct and kills the `issuedBy`-domain type-collapse inference hazard.
- ADR-0004 manifest-hash-over-meaning-bearing-artifacts (not raw two-file bytes) is the right release identity; whitespace/comment-stable.
- `EvidencePacket` `sh:or(containsClaim | isDerivedFrom)` correctly replaces the always-true `sh:minCount 0` no-op and stops packets being mis-inferred as `Claim` via `supportedBy`'s domain.
- Timezone-offset SHACL pattern on `publishedAt`/`receivedAt` is a real point-in-time gain over bare `xsd:dateTime`.
- Explicit refusal to use graph size / inferred-triple count / fluent explanations as success metrics; P2 degradation-to-champion on disagreement is the correct safety posture.

## 3. Defects & gaps (most severe first)

**D1 — H7 names bitemporal times but mechanizes none; no cutoff predicate, no snapshot binding.**
(a) H7 lists occurrence/effective/publication/receipt/cutoff, but `core.ttl` has `publishedAt`/`effectiveAt`/`receivedAt` with *no domain* and no `decisionCutoffAt`/`snapshotId`; `SemanticSnapshot` (H8) has no temporal binding. (b) Cutoff leakage (a fact with `receivedAt > cutoff` used in a decision) is *the* canonical look-ahead failure — it inflates backtests and produces untradeable live signals. Without a governed cutoff predicate, H12's cutoff-leakage test is nominally green but semantically vacuous. (c) In H7 declare `tw:decisionCutoffAt`, `tw:occurredAt` (OWL-Time `time:Instant`-aligned), `tw:snapshotId`, each with explicit domain + `sh:maxCount 1`; bind `SemanticSnapshot` to a cutoff. Instance validation rejects any admitted fact where `receivedAt > decisionCutoffAt`. **Do NOT** add a universal `effectiveAt ≥ publishedAt` or `occurred-before-published` axiom — guidance is future-effective and corrections describe the past; `publishedAt ≤ receivedAt` is also unsafe (embargoed/backfilled data). Gate on **receipt-vs-cutoff only.**

**D2 — Correction/supersession lineage named but not modeled or constrained.**
(a) H7 says "correction/supersession lineage" with no property, activity, or shape. (b) A restated EPS / amended filing that doesn't chain to the record it supersedes creates a silent parallel universe: two contradictory Claims coexist, append-only invariant (§6.1/§10) is nominally honored but the graph can't tell one invalidates the other, and retrospective root-cause classes "temporal leakage"/"data quality" become unreliable. (c) Add an H7 subtask: model `tw:CorrectionActivity a prov:Activity` with mandatory `prov:used` → superseded `SourceRecord`/`Claim` and `prov:generated` → correcting record, plus `prov:wasRevisionOf` on the new record and `prov:invalidatedAtTime`. SHACL shape rejects a correction lacking a superseded target and rejects in-place mutation of the superseded fact (preserve both; the snapshot binds which was known-at-cutoff). Add a *separate* `tw:ForwardGuidanceRecord` shape that explicitly *allows* `effectiveAt > receivedAt`, so future-effective guidance is not forced through the correction shape.

**D3 — Staging inversion: H8 before H9, H10 before H11; no `owl:imports` DAG.**
(a) `InstrumentBeliefState`/`Contradiction` (H8) must cite world events + exposure paths (H9); marker corroboration policy (H10) depends on evidence + events; portfolio (H11) scopes concentration corroboration. Modules are "own task, own tests" but there is no declared import graph or per-module versionIRI. (b) Building belief-state before the events it references forces forward-refs or makeshift duplicated temporal predicates that diverge from H7 — breaking manifest hashing (H4) and projection parity (H13), and letting a `InstrumentBeliefState` query silently use the wrong cutoff. (c) Add an **H0 (before H6): module dependency manifest** — machine-readable `owl:imports` DAG + per-module ontology IRI/versionIRI recorded in `SemanticReleaseManifest`; build step fails if a module references a term outside its import closure. **Reorder to H6 → H7 → H9 → H8 → H11 → H10**, and split belief-state into an H8b that imports H9.

**D4 — H6 identity has no false-merge / governed-identity-assertion shape; `owl:sameAs` hazard unaddressed.**
(a) H6 lists "identity-assertion status/confidence" but no constraint; the only false-merge negative test (H5) targets P0's flat model, not H6's role/alias/broker-instrument model. (b) Merging two distinct `Security` nodes (ticker reuse after delisting, dual NSE/BSE listing, ADR vs common share, broker-instrument alias) collapses distinct economic exposures — directly violating §6.2 ("cannot be summed without proving lifecycle and identity relationship") and corrupting sizing/concentration at the `DecisionFeatureSet` seam. (c) Add `tw:IdentityAssertion` (status `candidate|asserted|confirmed|disputed|rejected|superseded`, confidence, validity interval, provenance); SHACL rejects two distinct Securities asserted equivalent without a governed `IdentityAssertion`. **Never use `owl:sameAs`/functional properties for identity or broker-instrument linkage** — open-world `sameAs` propagates merges transitively and irreversibly across the projection; use a typed, revocable `tw:mapsToInstrument` object property with a validity interval instead. Add a SPARQL-constraint negative test: one broker instrument mapped to two Securities in overlapping intervals must fail.

**D5 — "Tradability" is bundled into H6 identity but never modeled.**
(a) H6 is titled "Identity/tradability" yet lists no account-specific `TradabilityRiskPacket`, venue segment, orderability, restrictions, circuits/bands, settlement, or data entitlement. (b) The spec is emphatic that the broad discovery universe and the account tradable allowlist "are never the same collection by accident" (§4, §8); conflating them risks acting on an untradeable instrument. (c) Split H6 into **H6a Identity canonicalization** (D4) and **H6b Tradability/Capability**: model `VenueSegment`, `ListingStatus`, `BrokerInstrument`, `AccountCapabilityAssignment`, `TradabilityRiskPacket`, `Restriction`, `CircuitBand`, `DataEntitlement`, each account/snapshot/policy-scoped and deny-by-default.

**D6 — PROV-O / OWL-Time reuse is nominal, not committed.**
(a) H7 says "PROV-O aligned" and the audit names OWL-Time, but `core.ttl` imports nothing and no term mapping exists. Undeclared alignment is not alignment. (b) Retrospective diagnosis (§12 "research extraction"/"data quality") and audit reproducibility require querying which agent/model/version produced an admitted assertion; a bespoke vocabulary won't interop and re-invents derivation and interval semantics (half-open vs closed interval boundary bugs are a real leakage edge — a role active "until T" vs "before T"). (c) H7b: `owl:imports <http://www.w3.org/ns/prov>`; `SourceRecord`/`Claim`/`EvidenceSpan`/`SemanticSnapshot` as `prov:Entity`; `ExtractionActivity`/`AdmissionActivity`/`ProjectionBuildActivity` as `prov:Activity`; extractor/model/human/policy as `prov:Agent`; use `prov:used`/`wasGeneratedBy`/`wasDerivedFrom`/`wasAttributedTo`/`wasRevisionOf`/`generatedAtTime`. H7a: `owl:imports <http://www.w3.org/2006/time>`, map validity/role/listing intervals to `time:Interval`/`hasBeginning`/`hasEnd` with a **documented half-open convention** and a sentinel (not null-defaults-to-now) for open intervals. Note (audit §5): PROV-O proves lineage, not correctness — keep admission status orthogonal to provenance.

**D7 — H8 evidence/reasoning lacks source-independence and admission semantics; belief-state misplaced.**
(a) H8 has claims/assertions/contradictions but no source-family independence, evidence span/locator, extraction-model version, admission policy, assertion status, or contradiction *target*. `InstrumentBeliefState` sits ahead of the events (H9) and markers (H10) that feed it. (b) Agents will double-count syndicated/wire-copied stories as independent corroboration, and treat extracted claims as admitted facts — inflating conviction on a single underlying source. (c) Add `tw:SourceFamily`, `tw:EvidenceSpan`, `tw:ExtractionModelVersion`, `tw:AdmissionPolicyRelease`, `tw:AssertionStatus`, and `tw:contradictsClaim`/`contradictsAssertion` (a `Contradiction` must name an opposing target — reject self-contradiction and target-less contradiction). Every `AdmittedAssertion` must link to an admission activity + policy + source record/span. Split belief-state / `DecisionFeatureSet` projection into **H8b after H9/H10**, emitting categorical-only features gated by `DecisionFeatureActivation`.

**D8 — H9 doesn't enforce observed vs contractual vs causal-hypothesis at the shape level.**
(a) H9 says "distinguish observed vs contractual vs causal hypothesis" but proposes no class/enum or constraint. (b) If a causal hypothesis ("supply-chain disruption caused the revenue miss") is stored identically to an observed fact ("A supplies B per filing"), a research agent cannot separate them and an unverified causal claim crosses the typed `DecisionFeatureSet` boundary and amplifies narrative. (c) Model `tw:RelationshipAssertion` superclass with mandatory `tw:relationshipType` (enum `OBSERVED|CONTRACTUAL|CAUSAL_HYPOTHESIS|INFERRED`), `tw:assertionConfidence`, subject/object roles, provenance, and validity interval; SHACL rejects a `RelationshipAssertion` lacking `relationshipType`; competency test verifies `FILTER(?type = "CAUSAL_HYPOTHESIS")` separates hypotheses from contractual facts; policy = causal/inferred is risk-only unless corroborated + activated.

**D9 — H11 portfolio projection has no governed allowlist / closed shape; can leak custody-cash-order truth across the seam.**
(a) H11 projects holdings/reservations/exposure/concentration/currency but never says which predicates are projectable; §6.2 is explicit that "semantic projections are never custody, cash, order or fill truth." (b) A leaked cash-balance or open-order-quantity triple could be queried as authoritative, diverge from relational truth without reconciliation noticing, and violate "no number crosses the research seam." (c) Add an explicit **projection-allowlist** to H11: projectable = concentration *band*/exposure *category*/completeness-dimension enum/mandate metadata/policy-release IDs/partition metadata; relational-only = cash buckets, buying power, order quantities, fill prices, cost basis. Add `tw:PortfolioProjectionShape` with `sh:closed true` so any raw quantity/cash/price/fill literal on a projection node fails; add a shape forbidding household pooling of buying power/cash/orders across accounts (§5 invariant).

**D10 — P2 metrics are gameable: "incremental decision value" undefined; no rejection/demotion path.**
(a) H13 lists correctness/provenance/cutoff-safety/false-merge/latency/degradation/"incremental decision value" with no estimator, baseline, significance test, or units; H14 has no rejection path, no `DecisionFeatureDeactivation`, no cooldown, no requirement to retain the frozen manifest. (b) Undefined value + no null hypothesis is precisely where a bigger/slower/no-better graph gets promoted, and without a governed demotion a harmful feature needs a code change (not a policy release) to turn off — violating the immutable-release invariant. (c) Define `relational_champion_score` (precision/recall/p95 latency on the H12 pack) as the null hypothesis; challenger is eligible only if `recall ≥ champion` AND `precision ≥ champion − ε (ε ≤ 0.05)` AND **zero new** cutoff-leakage AND **zero new** false-merge positives AND ≥1 pre-specified metric improves by a meaningful, out-of-sample effect size on a frozen evaluation manifest. Incremental decision value = change in a sealed, out-of-sample retrospective outcome metric attributable to the feature — never in-sample query counts. H14: add `tw:DecisionFeatureDeactivation` (immutable release mirroring activation), require the frozen manifest + promotion decision stored as provenance *before* activation, a configurable cooldown via `PromotionPolicyRelease`, and automatic demotion-to-research-only on any safety-metric regression.

## 4. Missing negative tests
- Fact with `receivedAt > decisionCutoffAt` admitted into a snapshot → fail (D1).
- Forward-guidance record with `effectiveAt > receivedAt` → **pass** (positive fixture; guard against accidentally rejecting valid future-effective guidance) (D2).
- `CorrectionActivity`/correction lacking a superseded target (`prov:used`/`wasRevisionOf`) → fail; in-place mutation of a superseded fact → fail (D2).
- Two distinct Securities asserted equivalent (or `owl:sameAs`) without a governed `IdentityAssertion` → fail (D4).
- One broker instrument mapped to two Securities in overlapping validity intervals → fail (D4); ticker reuse after delist/relist merging across intervals → fail.
- Two `IssuerRole`s for different securities sharing one `playedBy` entity with overlapping validity → detectable via SPARQL constraint (D6).
- `roleValidUntil < roleValidFrom`, or an open interval encoded as null instead of a sentinel → fail (D6).
- Syndicated/same-`SourceFamily` copies counted as independent corroboration → fail (D7).
- `AdmittedAssertion` without admission activity+policy, or `Contradiction` with no opposing target / self-contradiction → fail (D7).
- `RelationshipAssertion` missing `relationshipType`; a `CAUSAL_HYPOTHESIS` feeding a `DecisionFeatureSet` without `DecisionFeatureActivation` → fail (D8).
- Portfolio-projection node carrying a raw quantity/cash/price/fill literal, or household view pooling buying power across accounts → fail `sh:closed` shape (D9).
- Module Turtle referencing an upstream term without declaring `owl:imports` → build fail (D3).
- Competency query lacking a positive golden, an expected-empty counterexample, AND a cutoff-leakage exclusion case → CI fail (H12/D10).
- `DecisionFeatureActivation` without a linked frozen evaluation manifest → build fail; promoted challenger regressing on a safety metric → auto-demotion integration test (D10).
- Challenger unavailable/stale but decision proceeds on graph answer instead of degrading to champion → fail.

## 5. Standards-compliance findings
- **RDFS:** domain/range on new props (`playedBy`, `issuesSecurity`, …) are *inference*, not validation — every operational "must/must-not" needs a matching SHACL shape. Add a lint rule flagging any property whose domain/range has no corresponding SHACL property shape (the ADR-0005 lesson generalized).
- **SHACL (closed-world):** good `sh:or` use; extend closed-world validation to *snapshots*, not vocabulary. Use `sh:closed true` for sealed contract/projection nodes (D9), not open research graphs. Tighten the degenerate `payload_hash` regex `^sha256:[0-9a-f]+$|^sha256:.+$` (second alternative accepts uppercase/garbage) to `^sha256:[0-9a-f]{64}$`. Give each `sh:or` branch its own `sh:message`. Apply the timezone pattern to `roleValidFrom`/`roleValidUntil`, which currently lack it.
- **OWL (open-world):** absence ≠ false — never encode operational "must have"/"forbidden" only in OWL; keep it in SHACL/relational. Ban `owl:sameAs`/functional-property identity merges (D4); use typed, governed, revocable assertions.
- **PROV-O:** map/import, don't imitate (D6): `prov:Entity/Activity/Agent`, `used/wasGeneratedBy/wasDerivedFrom/wasAttributedTo/wasRevisionOf/generatedAtTime/invalidatedAtTime`; provenance proves lineage, not correctness — keep admission status orthogonal.
- **OWL-Time:** adopt `time:Instant`/`time:Interval`/`hasBeginning`/`hasEnd` (prefer `xsd:dateTimeStamp` where TZ is required) with a documented half-open convention; don't hand-roll boundary logic (D6).
- **OWL 2 profile:** declare the target profile (DL is plausible given the role model) and run a bounded consistency/profile check in the build to catch unsatisfiable classes — but keep production evaluation on a bounded entailment profile (no unrestricted OWL DL at query time), and each competency query names its allowed entailments.
